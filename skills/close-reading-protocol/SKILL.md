# Close Reading Protocol v1

> 严谨阅读协议。适用于需要对长文本进行全文级理解的场景。

## 问题背景

LLM 存在 "Lost in the Middle" 现象：对长文本的开头和结尾关注充分，但对中间约 60% 的内容注意力衰减。这导致分析结论可能遗漏关键信息，产生"只看头尾就下判断"的抽样错误。

## 协议流程（强制顺序执行）

```
收到待读文本
    │
    ├── Step 1: 文本分块
    │
    ├── Step 2: 逐块精读（每块输出注释）
    │
    ├── Step 3: 总览合成
    │
    ├── Step 4: 举证分析（所有结论附原文引用）
    │
    └── Step 5: 自检复核
```

### Step 1: 文本分块

将全文按自然语义边界切块。每块长度不超过 2000 字。

```
分块规则：
- 优先按章节/自然段切分
- 无章节标记时，按段落空行切分
- 空行也不足时，按 1500-2000 字切割，尽量在句号处断开
```

分块完成后，列出总块数。例如：

> 全文共分 12 块。开始逐块阅读。

### Step 2: 逐块精读

**每次只读一块。读完一块后才能读下一块。**

每块的输出格式：

```
## 块 N/N

【摘要】（2-3 句话概括本块核心内容）

【关键要素】（出现的角色、地点、时间信息、事件节点）

【疑点/待验证】（需要与后续块交叉核对的信息，没有则填"无"）

【原文摘录】（本块中印象最深或最具分析价值的 1-3 句原文）

────────────────────────────────
```

**禁止以下行为：**
- ❌ 一次读完所有块再统一输出
- ❌ 跳过中间块直接读开头和结尾
- ❌ 合并多个块输出单一"总摘要"

### Step 3: 总览合成

所有块阅读完毕后，合成全文总览：

```markdown
## 全文总览

### 结构
- 章节/段落分布：[块1-3：引子，块4-8：发展，块9-12：高潮+收束]
- 叙事节奏：[整体节奏评估]

### 角色动向
- [角色A]：在块2/5/9-11出场，弧线：...
- [角色B]：在块1/3/6/12出场，弧线：...

### 关键节点
| 节点 | 位置 | 说明 |
|------|------|------|
| 转折点1 | 块4 | ... |
| 高潮 | 块10 | ... |
```

### Step 4: 举证分析

分析中的所有诊断结论，必须附带原文引用。

```
✅ 正确示范：
"第5章中角色A说'我再也不会相信你了'（块6，原文），
结合第2章他同样对角色B说过类似的话（块2，原文），
表明角色A存在'信任创伤'这一人物核心弱点。"

❌ 错误示范：
"角色A有信任问题。"
（无原文引用 → 无效诊断）
```

**引用规范：**
- 每条结论至少引 1 处原文
- 标注所属块号
- 对比型结论（如"前后矛盾"）必须引两处原文

### Step 5: 自检复核

分析完成后，执行自检：

```
□ 所有块是否都已阅读？（检查块计数）
□ 是否有结论未附带原文引用？
□ 中间块（去掉首尾各 20%）的引用数量是否不低于总引用数的 30%？
   → 如果低于 30%，说明可能跳读了中间内容，请回到中间块补充阅读。
□ 是否有"感觉像"、"好像有"这类模糊表述？如有 → 定位到具体原文再确认。
```

所有检查项通过后，输出：

> ✅ 阅读协议执行完毕。全文 N 块已逐块精读，分析结论均有原文支撑。

## 被父 skill 调用时的使用方式

父 skill（如 mckee-story-analyzer 或 editor agent）在需要深度分析时，在分析流程中插入：

> 在分析前加载 close-reading-protocol，按协议执行 Step 1-5。
> 分析中的每一条诊断结论均须附带原文引用标注。

## 独立使用命令

当用户直接要求分析文本时（不经过父 skill），直接加载本协议全文执行。
